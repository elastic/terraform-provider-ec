# Coding Standards

This document outlines the coding standards and conventions used in the cloud provider
(`terraform-provider-ec`). Unlike the stack provider (`terraform-provider-elasticstack`),
this repository is **already 100% on the [terraform-plugin-framework](https://developer.hashicorp.com/terraform/plugin/framework)** —
there is no SDKv2 code and no migration in progress.

For where code lives see [`./repo-structure.md`](./repo-structure.md); for day-to-day commands
see [`./development-workflow.md`](./development-workflow.md).

## General Principles

- Write idiomatic Go.
  - [Effective Go](https://go.dev/doc/effective_go)
  - [Code Review Comments](https://go.dev/wiki/CodeReviewComments)
  - The [Google Style Guide](https://google.github.io/styleguide/go/index#about)
- Every Go file carries the Apache 2.0 license header (see [License headers](#license-headers)).
- The final state for a resource is derived from a **read following the mutative request**. `Create`
  and `Update` call the resource's internal `read` after the API call rather than building state from
  the create/update response (see `ec/ecresource/extensionresource/create.go`).

## Resource and data-source layout

Each resource is its own package under `ec/ecresource/<name>resource/`, and each data source under
`ec/ecdatasource/<name>datasource/`. The plugin-framework lifecycle is split across files, e.g.
`ec/ecresource/extensionresource/`:

- `schema.go` — the `Schema` method, the Terraform model struct (`modelV0`, one `tfsdk:"..."` field
  per attribute), the interface assertions, and `Metadata`/`Configure`/`ImportState`.
- `create.go`, `read.go`, `update.go`, `delete.go` — the CRUD methods. `read.go` defines an internal
  `read` helper reused by the create/update paths.
- `resource_test.go` and colocated `*_test.go` — unit tests; `testdata/` holds Terraform fixtures.

Larger resources (e.g. `ec/ecresource/projectresource/`) split models and sub-concerns into their own
files (`elasticsearch.go`, `security.go`, `observability.go`, `import.go`, `mocks.gen.go`).

Conventions to follow:

- Declare interface assertions so the compiler enforces the framework contract, e.g.
  `var _ resource.ResourceWithConfigure = &Resource{}` (see `extensionresource/schema.go`).
- Wire the API client in `Configure` via `internal.ConvertProviderData`; guard every CRUD method
  against an unconfigured client before use.
- Set `TypeName` in `Metadata` as `request.ProviderTypeName + "_<name>"`.
- Data sources follow the same shape (`DataSource` struct, `Schema`/`Read`/`Metadata`/`Configure`,
  a `modelV0`, and a `modelToState` flattener) — see `ec/ecdatasource/trafficfilterdatasource/datasource.go`.

### Shared helpers

Prefer the shared `ec/internal` packages over ad-hoc code:

- `ec/internal/converters` — conversions between cloud-sdk-go `models` and framework `types`
  (e.g. `ModelsTagsToTypesMap`).
- `ec/internal/planmodifiers` — reusable plan modifiers (`StringDefaultValue`, `AttributeChanged`,
  `UseStateUnlessTemplateChanged`, …).
- `ec/internal/validators` — reusable validators (e.g. `IsURLWithSchemeValidator`).

Resource-specific plan modifiers, validators, and custom types that aren't reused belong inside the
resource package (e.g. `useUnknownIfOtherChanges` in `extensionresource/schema.go`). Prefer schema-level
validation and plan modifiers over ad-hoc checks inside CRUD methods.

### Generated code

Code under `ec/internal/gen/serverless` is generated by `oapi-codegen` and carries a
`DO NOT EDIT` marker — **never hand-edit it**. Change the inputs and regenerate; see
[`./generated-clients.md`](./generated-clients.md).

## License headers

Every Go file must start with the Apache 2.0 (ASL2) header
("Licensed to Elasticsearch B.V. under one or more contributor license agreements…"). Headers are
managed by `go-licenser`:

- `make license-header` (part of `make format`) inserts missing headers.
- `check-license` (part of `make lint`) fails the build if any header is missing.

## Formatting and lint gates

Run `make format` before committing and `make lint` to gate. Both are thin wrappers over the
`build/Makefile.format` and `build/Makefile.lint` fragments.

`make format` runs: `license-header` → `golangci-lint run --fix` → `tfproviderlint -fix` →
`terraform fmt -write`.

`make lint` runs, and must pass:

- **`golangci-lint`** (config `.golangci.yaml`, v2) — enabled linters: `errcheck`, `ineffassign`,
  `misspell`, `staticcheck`, `unconvert`, `unused`, `govet`; the `gofmt` formatter enforces Go
  formatting. `go-licenser` is also run here.
- **`check-license`** — Apache 2.0 header presence.
- **`tfproviderlint`** — Terraform provider best-practice checks (the `/gen/` tree is excluded).
- **`tfproviderdocs`** — generated-docs consistency (see [`./documentation.md`](./documentation.md)).
- **`terraform fmt -check`** — formatting of `.tf` example/test files.

## Error handling

- Surface errors through `diag.Diagnostics`: call `resp.Diagnostics.AddError(...)` and bail early when
  `resp.Diagnostics.HasError()`.
- Detect "not found" with `errors.As` against the typed cloud-sdk-go error and call
  `response.State.RemoveResource(ctx)` so Terraform reconciles a deleted remote resource (see
  `extensionresource/read.go`).
- `errcheck` is enforced; handle or explicitly ignore returned errors (a narrow ignore list lives in
  `.golangci.yaml`).

## Testing

- Write **table-driven unit tests** with `t.Run()` subtests, using
  [`testify`](https://github.com/stretchr/testify) (`assert`/`require`) for assertions. Tests are
  colocated with the code they cover (e.g. `ec/internal/converters/convert_tags_test.go`).
- Unit tests (`make unit`) need no credentials and are always safe to run.
- **Acceptance tests hit the real, paid Elastic Cloud API**, take up to ~2 hours, and run out-of-band
  on CI — do **not** run them locally or from an agentic workflow. There is no local Docker stack.
  See [`./testing.md`](./testing.md) for the full contract.

## Changelog

User-facing changes require a per-PR entry at `.changelog/{PR}.txt` (the PR number, not a shared
CHANGELOG file). See [`./contributing.md`](./contributing.md) for the entry format.

## GitHub Agentic Workflow authoring (`imports:`)

> Applies once the agentic workflows land — see the note in [`../../AGENTS.md`](../../AGENTS.md).

As agentic workflows are added under `.github/workflows/` in later phases of the LLM-driven SDLC epic,
shared workflow fragments follow an **extract-on-second-use** convention: write a block inline the first
time, and the moment a *second* workflow needs the same block, factor it into a shared fragment and pull
it in via a [gh-aw](https://github.com/githubnext/gh-aw) `imports:` entry rather than copy-pasting. This
keeps prompts and steps single-sourced without prematurely abstracting one-off blocks.
